### New bug patterns (add to playbook)

10. **Radix Portal popups behind dialog backdrop** — Tailwind's default `z-50` is much smaller than this project's custom `z-modal-backdrop` (1040). Any portal popup (`SelectContent`, `PopoverContent`, etc.) opened inside a `Dialog` will be visible BUT click-intercepted by the dialog overlay. Fix: bump portal popups to `z-popover` (1060) or higher. Audit other portal components.
11. **API submit requires field that isn't in defaultValues** — react-hook-form omits keys whose values are `undefined` from the form `data`. If the backend DTO requires `isBoolean` on `isDefault`, the form must either (a) include `isDefault: false` in defaults, or (b) the submit handler must coalesce `?? false`. Pattern fix is in `ManageWhatsappTemplateModal.onSubmit`.
12. **Function components inside Radix `asChild` slot** — Radix passes refs through `Slot`. If the child is a plain function component, React warns and the ref is lost. Always `forwardRef` for any reusable wrapper that will be passed as `asChild` (or to a Radix primitive whose original component is a `forwardRef`). Seen here in `Modal.tsx`'s `StyledOverlay` / `StyledContent`.
13. **Fragment in `.map()` without key** — `steps.map(s => <>...</>)` silently fails React's key check. Use `<React.Fragment key={...}>` when iterating with non-DOM siblings.

## Session 5 (targeted fixes — change lesson drawer + dirty form on mount)

Targeted both bugs flagged by Session 4. Both now fixed and verified live.

### Bug 1 — "Change" button on a student's lesson row silently does nothing

**Root cause.** Pattern #14 (stale closure in recoil updater). In
`TeachingServiceItem.tsx` and `TeachingService.tsx`, click handlers wrote
`setStudentData(prev => ({ ..., tableDrawers: { ...studentData.tableDrawers, ... }}))`
— spreading the *captured-at-render* `studentData.tableDrawers` instead of
`prev.tableDrawers`. Between render and click, the persisted recoil state
(`persistLocalStorage` effect) could rehydrate with stale values, so the
write overwrote `isOpenAssignCourse` back to `false` (or kept it false
inadvertently). The drawer flipped to `true` in the persisted state but the
reader (`<CreateTeachingService open={isOpenAssignCourse}>`) saw it close
to false on the very next tick.

**Fix.** Swap `studentData.tableDrawers` → `prev.tableDrawers` in every
`setStudentData(prev => ...)` updater. Four sites total:

- `apps/admin/src/pages/StudentDetail/components/TeachingServiceItem.tsx`
  (lines ~675, ~732 — Change Lesson + Add Lesson buttons)
- `apps/admin/src/pages/StudentDetail/components/TeachingService.tsx`
  (lines ~264, ~306 — Add Course button + drawer close handler)

**Before/after.** Before: click "Change" on a lesson row → recoil persisted
state shows `isOpenAssignCourse: true` but `<CreateTeachingService>` never
mounts in the DOM. After: click "Change" → drawer mounts immediately with
"Change Lesson" header, student name/email/phone, course + class
dropdowns, and lesson time slot field. Verified live at
`/student-record/1?userId=4`.

### Bug 2 — Class edit tab shows "*Unsaved changes" banner on mount

**Root cause.** Three layered issues that all marked the form dirty before
any user input. Playbook pattern #15 with extra wrinkles.

1. **`useFieldArray.replace()` always marks the array dirty.** In
   `RegularClassSchedulePeriods.tsx`, an effect auto-seeds a default period
   when the API returns an empty `periodsV2`: `if (fields.length === 0)
   replace([defaultPeriod])`. `replace()` from `useFieldArray` writes
   with `shouldDirty: true` internally — there's no option to override
   that. So as soon as the child mounted, the form was marked dirty.
2. **`BasicSetting.tsx` propagates with `shouldDirty: true`
   indiscriminately.** A `localForm.watch(values => form.setValue(...,
   { shouldDirty: true }))` subscription fires on *every* local-form value
   change, including mount-time programmatic writes (e.g. the
   price-options syncing effect that aligns `priceOptions[0].amount` with
   `tuition` on mount). Even after #1 was fixed, this kept the form
   dirty because mount-time programmatic writes (`periodRepeatCount`
   defaulting to `-1`, etc.) were propagated as user-driven changes.

**Fix.** Two coordinated changes — neither alone is sufficient.

- Extracted `buildDefaultRegularV2Period()` and
  `buildDefaultRegularV2LessonRepeatFormat()` into
  `apps/admin/src/utils/regular-class-schedule.utils.ts`. Both
  `RegularClassSchedulePeriods.tsx` (the child auto-seeder) and
  `Class/index.tsx`'s `setFormData` (the parent reset) now use them.
- `Class/index.tsx`'s `setFormData` pre-seeds `regularScheduleV2.periodsV2`
  with `[buildDefaultRegularV2Period()]` for `regularV2` classes when the
  API returned no periods. The parent's `reset({classes: [...]})` thus
  sets defaults and values to identical shapes — the child's
  `replace([...])` no longer fires (`fields.length` is 1, not 0).
- `BasicSetting.tsx`'s `localForm.watch` subscription now gates
  `shouldDirty` on `info.type === 'change'`. Programmatic `setValue`
  calls (`info.type === undefined`) propagate the value to the parent
  without marking dirty; user-driven `'change'` events still mark the
  form dirty.

**Before/after.** Before:
`/teaching-service/edit-course?tab=class` → page loads → `*Unsaved
changes` banner appears with no interaction (`isDirty: true`,
`dirtyFields: { classes: [true] }`, even though `_defaultValues` and
`_formValues` were structurally identical). After: page loads with
`isDirty: false`, `dirtyFields: {}`, no banner. Typing into the `quota`
input now correctly flips `isDirty: true` and shows the banner.

**16. `useFieldArray.replace([...])` always marks the array dirty.** Calls
to `useFieldArray.replace()` always propagate as `shouldDirty: true` — the
hook has no option to override that. If you use `replace()` in a
mount-time `useEffect` to auto-seed an empty array, the form will be
dirty before the user has typed anything. Fix: either (a) pre-seed the
array at the parent's `defaultValues`/`reset()` level so the child never
sees `fields.length === 0`, or (b) avoid `replace()` for seeding and use
`form.reset(seededValues, { keepDirty: false })`.

**17. `form.watch(cb)` subscriptions that re-propagate with `shouldDirty:
true` leak dirty marks from mount-time programmatic writes.** A common
pattern is `localForm.watch(v => parentForm.setValue(..., v, {
shouldDirty: true }))` to sync a sub-form. The `watch` callback fires for
*every* value change — including programmatic ones from sibling
`useEffect`s (e.g. defaulting fields on mount). Gate `shouldDirty` on
`info.type === 'change'` to limit propagation to real user input.

### New bug patterns spotted (add to playbook)

**18. react-select multi-value chip's Remove aria-label stringifies JSX from
`formatOptionLabel`.** When a Select is `isMulti` and uses
`formatOptionLabel` that returns JSX (e.g. a course icon plus label), the
selected chip's Remove button gets `aria-label="Remove [object Object]"`
because react-select's MultiValue builds the aria-label from the rendered
chip children. `getOptionLabel` does NOT override this. Fix: branch on
`formatOptionLabel(data, { context })` — return a plain string when
`context === 'value'`, JSX when `context === 'menu'`. Audit other
multi-select selectors: `LabelSelector`, `TextSearchMultiSelector`,
`StudentSelector` (Promotion), `FieldSelect` (StudentCRM).

**19. AG Grid v33 "cell data type is object" warning (`#48`) when a column's
`field` resolves to an object but only a `cellRenderer` is provided.** AG
Grid v33 auto-infers a cell data type from the first non-null row's value;
for `object`, it expects a `valueFormatter` for sorting/filtering/copy.
Fix: add `valueFormatter: () => ''` (or a stringified summary) on
object-valued columns when the `cellRenderer` already handles display.

**20. Duplicate AG Grid `colId` from mixing `field` and explicit `colId`
in the same column-defs array (`#273`).** A `{ field: 'phone' }` def
produces colId `phone`; a sibling `{ colId: 'phone', ... }` collides and
AG Grid silently renames the second to `phone_1` (and subsequent ones
`phone_2`, etc.). Audit duplicate `field`/`colId` pairs whenever two
columns render the same logical concept; either remove one or assign a
distinct `colId`.
